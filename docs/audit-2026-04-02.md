# Deep Technical Audit — 2026-04-02

Scope: full codebase review of `dowdiness/mdsp` (Phases 0-4 complete).
Method: line-by-line code tracing of all packages (`lib/`, `pattern/`, `browser/`, root).

## Summary

The codebase is well-engineered for its stage. The Finally Tagless architecture,
transactional topology editing, compile-time graph validation, and
zero-allocation audio path are sound design choices. No showstopper bugs were
found in the primary execution path.

The findings below concern gaps in the safety model — places where runtime
mutation, long-running state, or validation asymmetry could produce incorrect
behavior under specific conditions.

---

## Confirmed Defects

### D1. Delay parameter update accepts out-of-range values

- **Location:** `lib/graph.mbt:2082-2086`
- **Severity:** HIGH
- **Status:** Confirmed

When `SetParam(DelaySamples, value)` is applied at runtime, `exact_int_value()`
converts the Double to Int but no check enforces
`0 <= delay_samples <= node.delay_max_samples`. Compare with Biquad parameters
which are validated by `valid_biquad_graph_params()` at line 2072. The downstream
`DelayLine::set_delay_samples()` clamps internally, so the effect is silent
clamping — but the graph node's `delay_samples` field retains the invalid value,
creating a mismatch between recorded state and actual delay line state.

### D2. Control application is not atomic

- **Location:** `lib/graph.mbt:1774-1788`
- **Severity:** MEDIUM
- **Status:** Confirmed

`apply_controls_impl` validates all controls against a simulated copy (lines
1775-1781), then applies in a second loop (lines 1784-1786). If
`apply_control_impl` fails during the second pass, the result is swallowed by
`ignore()`. Some controls will have been applied, others not. No rollback.

In practice this is unlikely because the validation simulates on the same node
array, but the `ignore()` masks any divergence between validation and
application.

### D3. ADSR recursive tick pattern is fragile

- **Location:** `lib/env.mbt:104-119`
- **Severity:** MEDIUM
- **Status:** Confirmed (bounded at depth 2, not a stack overflow)

When `attack_ms <= 0.0`, the Attack branch recursively calls
`tick_with_sample_rate()`, which enters Decay, which may recurse again into
Sustain. Maximum recursion depth is 2 (Attack -> Decay -> Sustain) because
Sustain does not recurse. Not a crash risk today, but the pattern is fragile —
a future stage transition that loops back would cause unbounded recursion. Should
be rewritten as an explicit loop.

## High-Risk Areas

### R1. Graph buffer indexing relies entirely on compile-time validation

- **Location:** `lib/graph.mbt:852-996` (non-feedback), `lib/graph.mbt:1040-1310` (feedback)
- **Severity:** HIGH (structural)

Every node in the process loop accesses `buffers[node.input0]` without runtime
bounds checking. Safety depends on: (1) `valid_reference()` at compile time,
(2) correct topological sort, (3) correct index map composition after
optimization. Hot-swap and topology editing introduce runtime mutation through
index remapping. One bug in `apply_topology_delete_node` or
`apply_topology_delete_chain` would cause silent out-of-bounds access.

### R2. Voice pool generation counter overflow

- **Location:** `lib/voice.mbt:236-238`
- **Severity:** HIGH (theoretical)

`slot.generation` and `self.next_allocation_order` are signed 32-bit integers
that increment without overflow protection. After 2^31 note_on calls on a single
slot, a stale `VoiceHandle` could match the wrapped generation. At maximum note
rate (~375 notes/sec with 128-sample blocks at 48kHz), overflow takes ~14 hours
per slot. The `allocation_order` counter (pool-wide) overflows faster with 32
voices.

### R3. Browser @ref.Ref globals and concurrent access

- **Location:** `browser/browser.mbt:71-195`
- **Severity:** HIGH (architectural)

~30 `@ref.Ref` globals hold graph state. No synchronization or documentation
of the required call ordering between main thread (`init_*`) and audio thread
(`process_*`). Whether WASM-GC memory access from AudioWorklet is serialized
depends on the JavaScript runtime, not on guarantees in this code.

### R4. Feedback edge indices not bounds-checked

- **Location:** `lib/graph.mbt:749-766`
- **Severity:** MEDIUM

Back-edge source/target indices from `remapped_feedback_edges()` are used to
index into `self_enabled[]`, `back_edge_input0_source[]`, etc. without verifying
`source_idx < node_count` and `target_idx < node_count`.

## Edge Case Vulnerabilities

### E1. Rational arithmetic overflow (Int64)

- **Location:** `pattern/rational.mbt:55-95`
- **Severity:** MEDIUM (acknowledged in code comments at lines 83-85)

All arithmetic operations use cross-multiplication that can overflow Int64.
Acceptable for musical patterns (small integer subdivisions), unsafe for
adversarial input. `Rational::new(Int64::min_value(), -1)` would overflow
in GCD's `abs()` call.

### E2. `note_name()` parser silently accepts invalid input

- **Location:** `pattern/control.mbt:23-73`
- **Severity:** LOW

Empty string -> MIDI 0. Unrecognized letter -> semitone 0. Trailing garbage ->
ignored. No error signal on malformed input.

### E3. Empty graph edge case in output copy

- **Location:** `lib/graph.mbt:995`
- **Severity:** LOW (mitigated by validation)

`buffers[buffers.length() - 1]` accesses index -1 if buffers is empty. Should
not happen due to compilation requiring an Output node, but no explicit guard.

### E4. `merge_control()` quadratic complexity

- **Location:** `pattern/control.mbt:109-120`
- **Severity:** LOW

O(n*m) nested loop. Acceptable for typical patterns, but no upper bound.

## Structural Observations

### S1. Flat DspNode struct trades type safety for locality

`lib/graph.mbt:85-155` — Single struct with generic fields for all 18 node
kinds. Easy to construct semantically invalid combinations. Mitigated by
constructors, but topology editing reconstructs nodes from raw fields.

### S2. Browser package is monolithic

`browser/browser.mbt` — 1200 lines, 6 copy-pasted graph variants, each with
its own globals. Changes must be manually replicated. Highest risk of
copy-paste bugs.

### S3. Validation split between compile-time and runtime

`lib/graph.mbt:1927-1982` (compile-time) vs `lib/graph.mbt:2040-2136`
(runtime). Different code paths with different checks for the same parameters.
D1 is a direct consequence of this split.

### S4. Pattern-DSP interface is stringly typed

`pattern/control.mbt` — ControlMap uses `Map[String, Double]`. No compile-time
verification that pattern keys ("note", "cutoff", "gain", "pan") match DSP
control slots.

---

## Lessons Learned

1. **Validation must be symmetric.** When two code paths can set the same
   parameter (compilation vs runtime control), they must enforce the same
   constraints. D1 exists because they don't.

2. **"Trust the compiler" breaks at mutation boundaries.** Static validation is
   sufficient for immutable graphs. Once hot-swap and topology editing mutate
   node arrays at runtime, the trust chain extends through index remapping — one
   remapping bug away from silent corruption.

3. **Long-running state needs overflow planning.** Generation counters and
   allocation order are cheap to increment but expensive to wrap. Production
   audio engines run for hours or days.

4. **Copy-paste scales linearly in bugs.** The browser package's 6 variants
   share logic that should be factored out. The next bug will come from a
   change applied to 5 of 6 variants.

5. **Stringly-typed interfaces hide mismatches.** ControlMap keys are invisible
   to the type system. A typo produces silence, not an error.
